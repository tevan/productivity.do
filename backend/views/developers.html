Developer API

Build integrations on top of productivity.do — read and write tasks, events, bookings, and subscribe to real-time webhook events.

Overview

The API is REST-style, JSON-bodied, and versioned. All v1 endpoints live under https://productivity.do/api/v1/.

Use it to:

  • Pull a calendar of upcoming bookings into your CRM.
  • Sync tasks with another to-do system.
  • Create events on a Google Calendar from a third-party app.
  • Receive a webhook the moment someone books a meeting.
curl https://productivity.do/api/v1/ping
# {"ok":true,"time":"2026-04-29T15:00:00.000Z","version":"v1"}

Authentication

All endpoints require a Bearer API key. Generate one in the productivity.do app under Settings → Developer → API keys.

Authorization: Bearer pk_live_xxxxxxxx.yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy

The full secret part is shown once at creation time. Store it securely. The visible pk_live_xxxxxxxx prefix is non-secret and used to identify the key in logs.

Don't ship API keys in client-side code. Treat them like passwords. Use the per-page public booking endpoints (/api/public/booking/...) for client-side widgets — those are explicitly safe to call from a browser.

Scopes

Each API key carries a list of scopes. Endpoints declare which scope(s) they need.

ScopeWhat it grants
adminWildcard — all scopes
read:tasks / write:tasksList and mutate Todoist-backed tasks
read:events / write:eventsRead and mutate Google Calendar events
read:calendarsList the connected Google calendars
read:bookings / write:bookingsRead booking pages and confirmed bookings
read:webhooks / write:webhooksManage outbound webhook subscriptions

Errors

Errors return a JSON body with ok: false and a human-readable error string. HTTP status maps roughly:

StatusMeaning
400Bad request (validation)
401Missing or invalid API key
403Key lacks the required scope
404Resource not found
409Conflict (e.g. slot no longer available)
429Rate limit exceeded
502Upstream (Google / Todoist) error

Rate limits

Default: 120 requests per minute per API key. Anonymous traffic (e.g. the public booking widget hitting /api/public/...) is limited per IP. X-RateLimit-Limit and X-RateLimit-Window headers are returned on every response.

If you need higher limits for a partner integration, get in touch.

OpenAPI spec

Machine-readable spec at /api/v1/openapi.json. Drop it into Postman, Insomnia, Stainless, or your codegen of choice.

Tasks

Tasks are backed by Todoist on the server side. You read and write them through the same shape as the in-app SPA. Each task carries Todoist-owned fields (content, priority, dueDate, …) plus productivity.do-only fields that don't sync to Todoist:

  • estimatedMinutes — per-task time estimate. Used by auto-scheduling.
  • localStatus — kanban board column. One of todo, in_progress, a custom_<n> key, or null (defaults to To Do). Done is NOT a writable status — to mark a task done, use POST /tasks/{id}/complete. Done = Todoist completed.
  • localPosition — manual sort order within a column.
GET /api/v1/tasks read:tasks
List active tasks. Response merges Todoist and local-only fields.
curl -H "Authorization: Bearer $TOKEN" https://productivity.do/api/v1/tasks
POST /api/v1/tasks write:tasks
Create a task. Optional estimatedMinutes and localStatus persist locally.
curl -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"content":"Pay invoice","priority":3,"dueDate":"2026-05-15","labels":["work"],"estimatedMinutes":15,"localStatus":"in_progress"}' \
  https://productivity.do/api/v1/tasks

Body fields: content (required), description, priority (1–4, where 4 is urgent), dueDate (YYYY-MM-DD), dueDatetime (ISO), labels[], projectId, estimatedMinutes, localStatus.

PUT /api/v1/tasks/{id} write:tasks
Update task fields. Pass projectId to move the task to a different project. Pass localStatus to move the card on the kanban board (no Todoist call). Set localStatus to null to clear it.
curl -X PUT -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"localStatus":"in_progress"}' \
  https://productivity.do/api/v1/tasks/$TASK_ID
POST /api/v1/tasks/{id}/complete write:tasks
Mark a task done. This is the canonical way to move a card to the Done column — Done reflects Todoist's completion flag, not a writable localStatus.
POST /api/v1/tasks/{id}/reopen write:tasks
Reopen a completed task. The task returns to whichever localStatus it last had (or To Do if none).
DELETE /api/v1/tasks/{id} write:tasks
POST /api/v1/tasks/bulk write:tasks
Bulk-create up to 100 tasks per request. Body: {"items":[{...},{...}]}. Returns per-item results so partial failures don't block the rest.

Task columns (kanban)

Each user has a customizable kanban board with up to five columns. Columns map to stable status keys; tasks reference those keys via localStatus. The default columns are To Do (todo), In Progress (in_progress), and Done (done). Users can rename columns and add up to two custom columns (custom_<n>) — but management is in-app only; the public API is read-only here.

GET /api/v1/task-columns read:tasks
List the calling user's columns, ordered left-to-right.
curl -H "Authorization: Bearer $TOKEN" https://productivity.do/api/v1/task-columns

Response shape:

{
  "ok": true,
  "columns": [
    { "id": 1, "position": 0, "name": "To Do",       "statusKey": "todo" },
    { "id": 2, "position": 1, "name": "In Progress", "statusKey": "in_progress" },
    { "id": 3, "position": 2, "name": "Done",        "statusKey": "done" }
  ]
}

Use statusKey when writing localStatus on a task. Don't depend on name being stable — users can rename columns at will.

Projects

Todoist projects. Create / rename / recolor / delete from the API.

GET/api/v1/projectsread:tasks
List projects.
POST/api/v1/projectswrite:tasks
Create a project.
curl -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"name":"New project","color":"berry_red","isFavorite":true}' \
  https://productivity.do/api/v1/projects

Body fields: name (required), color (Todoist color name like berry_red, blue, charcoal; see Todoist docs), parentId, isFavorite.

PUT/api/v1/projects/{id}write:tasks
Rename / recolor / favorite a project. Body: { name, color, isFavorite } (any subset).
DELETE/api/v1/projects/{id}write:tasks
Delete a project. Tasks in it are deleted by Todoist; you may want to move them first.

Sections

Sections are columns within a Todoist project (e.g. "Backlog / This week / Doing"). Tasks reference them via sectionId.

GET/api/v1/sections?projectId=…read:tasks
List sections, optionally scoped to a single project.
POST/api/v1/sectionswrite:tasks
Create a section. Body: { name, projectId, order? }.
PUT/api/v1/sections/{id}write:tasks
Rename a section.
DELETE/api/v1/sections/{id}write:tasks
Delete a section. Its tasks remain in the project but lose their section assignment.

Labels

Manage the label registry (rename, recolor, delete). Tasks reference labels by name in their labels[] array.

GET/api/v1/labelsread:tasks
List labels.
POST/api/v1/labelswrite:tasks
Create a label. Body: { name, color?, isFavorite? }.
PUT/api/v1/labels/{id}write:tasks
Rename / recolor / favorite a label. Renaming a label here renames it on every task that uses it, since Todoist labels are referenced by name.
DELETE/api/v1/labels/{id}write:tasks
Delete a label. Removes it from every task.
GET/api/v1/filtersread:tasks
List Todoist saved filters ("today | overdue", etc). Pro feature — returns an empty array on free accounts.

Quick add

Send a single string and let Todoist parse the date, label, priority, and project.

POST/api/v1/tasks/quickwrite:tasks
Create a task from natural-language text.
curl -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"text":"Buy milk tomorrow at 5pm @errands p1"}' \
  https://productivity.do/api/v1/tasks/quick

Returns the fully-parsed task. The p1 sets priority to urgent (Todoist's 4); @errands attaches that label; tomorrow at 5pm sets dueDatetime.

Bulk update

PUT/api/v1/tasks/bulkwrite:tasks
Apply patches to up to 100 tasks per request. Body: { items: [{ id, ...patch }] }. Each item may include projectId to move. Returns per-id results so partial failures don't block the rest.

Booking pages

Read booking pages and the confirmed bookings on them. Public-facing slot lookups and reservations live under /api/public/booking/... (no auth required) — see the Widget API below.

GET /api/v1/booking-pages read:bookings
GET /api/v1/booking-pages/{id} read:bookings
{id} may be the UUID or the URL slug.
GET /api/v1/booking-pages/{id}/bookings read:bookings
List confirmed bookings for the page (most recent first). Use ?limit=<n> up to 500.

Bookings

The owner-facing endpoints for managing booking pages (creating pages, defining availability, etc.) live under /api/booking-pages/... and require a logged-in session today. They will be promoted to /api/v1 in a future release; until then, prefer the public widget endpoints below for write flows.

Webhooks

Subscribe to events to be notified when bookings are made or tasks change.

GET /api/v1/webhooks read:webhooks
POST /api/v1/webhooks write:webhooks
Subscribe a URL to one or more event names.
{
  "url": "https://your-server.example.com/hooks/productivity",
  "events": ["booking.created", "booking.cancelled"]
}

Response includes a freshly-minted secret. Store it — you'll need it to verify signatures.

POST /api/v1/webhooks/{id}/rotate-secret write:webhooks
Rotate the signing secret. Subsequent deliveries are signed with the new value.
DELETE /api/v1/webhooks/{id} write:webhooks

Webhook event types

EventWhen it fires
booking.createdA new booking is confirmed.
booking.cancelledA booking is cancelled (by host or invitee).
booking.rescheduledA booking's time changes.
task.createdA task is created via the API.
task.updatedA task is updated.
task.completedA task is marked complete.
task.movedA task's localStatus changes (kanban board move). Payload: { id, fromStatus, toStatus }.
*Wildcard — receive every event.

Webhook payload format

Every delivery is a JSON POST with these top-level fields:

{
  "event": "booking.created",
  "deliveredAt": "2026-04-29T15:30:00.000Z",
  "data": {
    "page": { "slug": "30min", "title": "30-minute intro" },
    "booking": {
      "id": "...",
      "name": "Avery Patel",
      "email": "avery@example.com",
      "startIso": "2026-05-12T20:00:00.000Z",
      "endIso":   "2026-05-12T20:30:00.000Z",
      "timezone": "America/Denver"
    }
  }
}

HTTP headers on each delivery:

X-Productivity-EventThe event name.
X-Productivity-TimestampUnix milliseconds when the signature was generated.
X-Productivity-Signaturet=<ts>,v1=<hex>. The hex is HMAC-SHA256 over ${ts}.${rawBody} using your subscription secret.
X-Productivity-DeliveryUnique id for this delivery (idempotency).

Verifying signatures

// Node.js example
import { createHmac, timingSafeEqual } from 'crypto';

function verify(req, secret) {
  const header = req.headers['x-productivity-signature'] || '';
  const parts = Object.fromEntries(header.split(',').map(p => p.split('=')));
  const ts = parts.t, sig = parts.v1;
  if (!ts || !sig) return false;

  // Reject deliveries older than 5 minutes (replay protection).
  if (Math.abs(Date.now() - Number(ts)) > 5 * 60 * 1000) return false;

  const expected = createHmac('sha256', secret)
    .update(`${ts}.${req.rawBody}`)  // raw bytes, not parsed JSON
    .digest('hex');
  const a = Buffer.from(sig, 'hex');
  const b = Buffer.from(expected, 'hex');
  return a.length === b.length && timingSafeEqual(a, b);
}
Tip: Use the raw request body for the HMAC. If your framework parses JSON before you can see the bytes, capture the raw body in middleware first. The timestamp prevents replay attacks — reject any delivery with a stale t=.

Retry policy

If your endpoint doesn't return a 2xx within 8 seconds, we retry with exponential backoff: 1m → 5m → 30m → 2h → 12h. After five failed attempts the delivery is abandoned. You can inspect failed deliveries via GET /api/webhooks/{id}/deliveries in the admin app.
