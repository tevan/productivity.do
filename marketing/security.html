Security & Privacy

Your calendar is some of your most personal data. Here's exactly how we protect it.

Data isolation

Every user's data is tagged with a user_id at the database level. Every query is scoped to the authenticated user. Cross-tenant access is impossible by API design — there is no endpoint that lets you reference another user's data.

Authentication

  • Passwords are stored as bcrypt hashes (cost factor 10). Plaintext passwords are never logged or persisted.
  • API keys are stored as SHA-256 hashes. The plaintext secret is shown exactly once at creation; we cannot recover it. Comparison is timing-safe (crypto.timingSafeEqual).
  • Sessions use signed HTTP-only cookies with SameSite=lax. Sessions expire after 30 days of inactivity.
  • OAuth tokens for Google Calendar are stored encrypted at rest (AES-256-GCM with per-row IV).

Transport security

  • All connections require TLS 1.2+ via Cloudflare-issued origin certs.
  • HSTS headers enforce HTTPS for one year.
  • Strong-cipher suite only (ECDHE + AES-GCM / CHACHA20-POLY1305).

Webhook signing

Outbound webhooks are signed with HMAC-SHA256 over ${'{ts}'}.${'{body}'} and delivered with the timestamp in X-Productivity-Signature. Receivers should reject deliveries older than 5 minutes (replay protection). Secrets can be rotated without downtime via the POST /api/webhooks/:id/rotate-secret endpoint.

Server-side request forgery (SSRF) protection

User-supplied webhook URLs are validated before persistence and again at delivery. We allow HTTPS only and reject URLs targeting loopback, RFC1918, CGNAT, and IPv6 internal ranges.

Race-condition hardening

Booking creation is atomic — slot availability re-check, single-use invite redemption, and the booking insert all run inside one SQLite transaction. Concurrent requests for the same slot cannot both succeed.

Backups & durability

  • SQLite WAL mode for crash safety.
  • Three-tier backup: hourly off-site (encrypted), every-15-minute local, plus daily long-term.
  • Restore drills run quarterly.

What we don't do

  • We don't sell your data, ever.
  • We don't train machine-learning models on your calendar content.
  • We don't share your data with third parties (other than the integrations you explicitly authorize, like Google Calendar).
  • We don't use third-party analytics or trackers on signed-in pages.

Compliance

  • GDPR: data export and deletion via /account/export and /account/delete. EU customers can request a Data Processing Agreement.
  • CCPA: California residents have the same rights via the same flows.
  • SOC2 Type II: in progress for Team customers (target 2026 Q4).

Reporting a vulnerability

Found a security issue? Email security@productivity.do. We respond within 24 hours and disclose responsibly. Bug bounty program coming soon.

Open-source security

Our code review process surfaces security findings publicly via signed commits. Recent hardening passes are documented in our changelog. The backend will be open-sourced under a source-available license.
